## Problem Statement

VentureX has shipped candidates (M13 — the X-axis: 53 named competitors with rationales + web evidence) and parameters (M14 — the Y-axis: 51 fact-collection columns split across Tier 1 Universal / Tier 2 Framework / Tier 3 Dynamic). The competitor comparison table is now an **empty 53 × 51 matrix = 2,703 cells**. M15's job is to fill those cells with researched, cited facts that the **Innovera analysis engine can consume** as input.

This is the load-bearing step that makes VentureX a real Competely replacement rather than a candidate list.

---

## Status Quo

Innovera's current upstream is a webresearch agent that produces "~2 pages of brittle, shallow output" per venture. Innovera's analysis engine is excellent — but garbage in, garbage out. The downstream is starved for grounded, structured input.

When consultants build competitor dossiers by hand today, two failure modes dominate:

1. **Task-size shortcut.** A 51-column × 53-row table is overwhelming; consultants end up filling the easy cells and skipping or approximating the rest. The output ends up time-boxed to "what we can finish," not "what the analysis needs."
2. **LLM verification overhead.** When LLMs are used to fill cells, hallucination risk drives consultants to verify every claim. Verification time often exceeds the time the LLM saved. The net win is small or negative.

M15 has to address both: do the lifting consultants can't do at scale, AND produce output trustworthy enough that verification is fast.

---

## Premises (all five confirmed in Phase 3)

1. **Innovera analysis engine is the primary downstream consumer.** M15 output must fit what Innovera ingests, not just look good in a UI. UI is secondary; data quality and export shape are primary.
2. **Citation discipline > cell-fill quantity.** Source URL + snippet attached to every claim is the verification fast-path. A cell with a citation a consultant can click and confirm in 5 seconds beats a cell with a more "complete" value but no source.
3. **Verification cost dominates research cost.** Target metric: "consultant can verify a 51-cell dossier in <15 minutes." Not "produce a full table without consultant review."
4. **Depth-first per-candidate dossier is the wedge** (Q4 locked). Nail one Schneider Electric dossier end-to-end before scaling. Once one works, the rest is parallelization, not architecture.
5. **Cells need a tri-state: `verified | inferred | unknown`.** Hiding uncertainty as "researched fact" hides hallucination risk. Consultants triage on confidence; Innovera analysis weights cells on confidence.

## Approaches Considered

### Approach A: Per-candidate batched call (minimal viable)
One Opus call per candidate, all 51 params in one prompt. Input: profile + candidate name/rationale/M13 citations + 51 param prompt_hints. Output: 51 cells with model-picked citations from M13 evidence + training-data citations.
- **Cost:** ~$10 per venture (53 calls × $0.20)
- **Effort:** human ~6–8h / CC ~75–120min
- **Pro:** Simplest, reuses M13 patterns, fastest first dossier
- **Con:** Attention dilution risk with 51 cells per call; citation rigor depends on model self-discipline with no search-layer enforcement
- **Completeness:** 7/10

### Approach B: Per-cell Exa+LLM (ideal architecture)
Every (candidate, param) pair gets its own Exa search shaped by param's prompt_hint, plus a small Haiku/Sonnet extraction call from the snippet.
- **Cost:** ~$30–70 per venture (2,700 Exa @ $0.01–0.025 ≈ $27–68 + 2,700 Haiku extractions @ $0.002 ≈ $5; assumes snippet cap of top-3 results × ~500 tokens each). **Note:** spec-review redid this math; B is cheaper than the initial doc suggested. The case for C rests on **complexity and latency control**, not raw cost.
- **Effort:** human ~12–16h / CC ~3–4h
- **Pro:** Citation drives fact (premise P2 maximally honored); minimum hallucination
- **Con:** Two-stage pipeline (Exa → Haiku) for every cell adds orchestration complexity even where the cell doesn't need it (e.g., `founded_year`); ~30–60min latency per venture sequential, with rate-limit concerns at scale; Tier 1 universal cells get treated the same as venture-specific differentiators (no investment-to-value matching).
- **Completeness:** 10/10

### Approach C: Hybrid tiered (RECOMMENDED — chosen)
Match research investment to cell value.
- **Tier 1 universal (15 params × 53 = 795 cells)** — one batched Opus call per candidate, training-data-only. Facts like "founded year" and "HQ location" rarely change; Exa search per cell would be overkill.
- **Tier 2 framework (21 × 53 = 1,113 cells)** — batched Opus call per candidate using M13 evidence. Citation discipline inherits from M13's existing grounding work.
- **Tier 3 dynamic (15 × 53 = 795 cells)** — per-cell Exa + Haiku extraction. The venture-specific differentiators that justify M15 vs Competely live here; they get the deepest research.
- **Total cost:** ~$30–80 per venture
- **Effort:** human ~10–12h / CC ~2–3h
- **Pro:** Matches investment to value; failure modes contained to one tier at a time
- **Con:** Three orchestrator paths instead of one; tier-handoff logic is the most complex code in M15
- **Completeness:** 9/10

### Approach D: Agentic
Give Claude a tool to call Exa, let it research cells autonomously per candidate.
- **Cost:** $20–200 per venture (variable)
- **Effort:** human ~15–20h / CC ~4–6h
- **Pro:** Handles unknown unknowns
- **Con:** Hardest to constrain (agent loops notoriously unbounded); hardest to test; least predictable cost; overkill for V1
- **Completeness:** Hard to score — depends on guardrails

## Recommended Approach: C (Hybrid Tiered)

```
                M15 — TIERED CELL RESEARCH DATA FLOW
                ─────────────────────────────────────

  /ventures/[id] in status='parameters_ready'
       │ user clicks "Research candidate dossiers"
       ▼
  server action: triggerStage5CellResearch(ventureId)
       │
       ▼
  ┌──────────────────────────────────────────────────────────────────┐
  │  src/server/stage5-cells.ts                                       │
  │                                                                    │
  │  1. claimParametersReadyStatus (atomic, mirror P3-D5)              │
  │  2. load latest candidate set + latest parameter run               │
  │  3. for each candidate (depth-first, ONE at a time for V1):        │
  │      a. Tier 1 call: batched Opus, training-data-only              │
  │         → 15 cells with low-citation expectation                   │
  │      b. Tier 2 call: batched Opus + M13 evidence                   │
  │         → 21 cells with M13-citation inheritance                   │
  │      c. Tier 3 loop: per-param Exa search → Haiku extraction       │
  │         → 15 cells with fresh Exa citation per cell                │
  │      d. insert 51 cells with confidence tri-state                  │
  │  4. mark status='cells_ready'                                      │
  └──────────────────────────────────────────────────────────────────┘
       │
       ▼
  /ventures/[id]/dossier/[candidate] page (minimal UI)
  → consultant verifies first dossier before authorizing full-scale run
  → "Approve all 53" button triggers parallel Tier 1+2+3 for remaining 52
```

### Why tiered

- **Tier 1 (universal)** facts have known-stable answers. A "founded year" cell doesn't need a fresh web search; the LLM's training data is fine. Cheap path for the 31% of cells that fall here.
- **Tier 2 (framework)** facts overlap with what M13 already researched. Reusing the M13 citation block as evidence keeps marginal cost low while inheriting M13's grounding discipline.
- **Tier 3 (dynamic)** facts are the venture-specific differentiators. These are where Competely fails — and where M15 wins by going deep. Per-cell Exa is justified here.

### Wedge: one Schneider Electric dossier first

V1 ships with a "research one candidate" affordance, not "research all 53." Consultant approves the Schneider dossier before authorizing parallel research on the remaining 52. This honors P3 (verification dominates) and contains the cost-blast-radius if Tier 3 misbehaves.

### Tier 3 fallback chain (added post-review)

When Exa returns zero results for a Tier 3 cell, the orchestrator does NOT fall through to a fabricated answer. The chain:
1. **Retry once** with a broadened query (drop the most specific keyword from `prompt_hint`).
2. **If still empty:** store cell as `confidence='unknown'`, value=null, citation=null, reason='no_evidence_found'. The cell is honest about its gap rather than fake.
3. **Never call Opus to fill a Tier 3 cell from training data.** Tier 3 exists specifically for facts that need fresh evidence; bypassing the search defeats the architecture.

### Concurrency model (added post-review)

- **Across candidates**: V1 ships sequential (one candidate at a time) to keep the wedge clean. Once the Schneider dossier validates the pipeline, M15.1 introduces parallelism with a cap of 5 candidates in flight.
- **Within a candidate**: Tier 1 and Tier 2 batched calls run sequentially (one Opus call each). Tier 3's 15 per-cell Exa+Haiku pairs run **in parallel with a cap of 3 concurrent Exa calls** (respects Exa's rate-limit headroom). Total per-candidate latency: ~6–8 min when Tier 3 is parallelized 3-wide.
- **Idempotency**: Cells are written incrementally per-tier as each tier completes. If the orchestrator crashes after Tier 1 but before Tier 2, restart resumes from Tier 2 for that candidate (the `cells` table acts as the resume checkpoint). Status transitions are atomic per candidate, not per venture.

## Open Questions (resolve in `/plan-eng-review`)

- **Cell storage shape**: jsonb on candidate_companies (atomic with candidate row) vs new `cells` table (candidate_id × parameter_id × value + citation + confidence). The latter is cleaner for Innovera export and per-cell queries; the former is simpler but loses sort/filter flexibility. Lean: separate `cells` table.
- **Confidence tri-state encoding**: `enum('verified', 'inferred', 'unknown')` vs `numeric 0.0–1.0` with thresholds. The enum is more honest about source quality; the numeric matches Stage 4 deferred-scoring pattern (P3-D15). Lean: enum, with the option to add numeric later.
- **Innovera ingest contract**: what does Innovera actually expect? JSON shape, CSV, API call? This needs a 15-minute conversation with the Innovera team before M15 codes the export step.
- **Consultant verification UI**: minimal (read-only with citation links) vs editable (cell-level overrides + audit trail). Lean: minimal in V1; editability is M16 polish.
- **Cell re-research affordance**: when a consultant marks a cell "wrong," can they trigger a re-research with corrective feedback (mirror the P3-D21 retry pattern)? Lean: yes, this is exactly the M14 retry pattern generalized.
- **Cost cap**: M14's run budget is $5 (D4). M15 at $30–80 per venture blows past that. Need a new budget tier for M15 — or do we widen D4? Lean: per-venture budget cap that's separate from per-run.

## Success Criteria

1. **First dossier beats the bar.** One Schneider Electric dossier with 51 cells. Tier 1 universal cells (15) carry training-data values with NO citation requirement (these are stable identity facts). Tier 2 + Tier 3 cells (36 combined) must hit ≥85% cited and ≥70% verified-tier confidence. End-to-end production <30 minutes; consultant verification of the 36 cited cells <15 minutes.
2. **Cost in band.** Per-venture cost lands in $30–80, with Tier 3 dominating spend.
3. **Innovera ingests cleanly.** The export format works first-try when fed to the Innovera analysis engine — no schema gymnastics required.
4. **Hallucination rate measurable and low.** Of cited cells, <5% have citations that don't actually support the value when a consultant clicks through.
5. **Re-research works.** When a consultant marks a cell wrong, the corrective-feedback retry produces a better cell on attempt 2 (mirror of the M14 P3-D21 pattern).
