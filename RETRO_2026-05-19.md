## What Claude missed

- **Didn't probe `src/server/` for existing Stage 4 files before drafting the sprint plan.** A 30-second `ls src/server/` on day one would have caught `stage4-parameters.ts` and prevented the off-target planning pass.
- **Didn't run `git status` early enough.** Untracked files (`parameter_builder.md`, the parameter-builder code) were sitting there the whole time. First saw them at the orchestrator step, hours in.
- **Took PHASE3.md §2 M14 row at face value.** Spec docs go stale when parallel work is in flight. Should always cross-check the spec against the tree before treating it as authoritative.

## What worked

- The CEO + eng review process produced real architectural decisions that survive the milestone shift. P3-D18 (separate `aggregate_score` column) and P3-D21 (single SQL UPDATE atomic write) generalize beyond this specific milestone — they're correct patterns for any future "compute and persist many rows of derived data" stage.
- Strict Zod refinement pattern (P3-D19's `makeStrictStage4ScoringOutputSchema` factory) is a reusable defensive pattern for any LLM call where output set must match input set 1:1.
- M13's §6b reframe (P3-D13) is the clean win of the day. Venture-agnostic acceptance criteria are now the precedent for future spec work.
- Migration 0005 ran cleanly; the columns are dormant but available if a future milestone wants Likert scoring layered on top of parameter-collected facts.

## Lessons

1. **Probe the codebase before drafting a sprint plan.** Skill's Step 0 ("what already exists?") should explicitly include `ls src/server/ src/app/` + grep for stage-named files. Adding to the personal playbook for next time.
2. **Spec ≠ reality.** PHASE3.md is a planning artifact, not a state file. Cross-check against `git status` + `ls` + recent commits before believing it.
3. **Surface divergences early; cleanup is cheaper than continuing.** Caught this at the orchestrator step, not after writing the orchestrator and UI. The cost of a 30-min cleanup pass is dramatically less than the cost of two parallel competing implementations.
