# Claude Table — Comparison Table Viewer Spec

**Owner:** Harry · **Stack:** Next.js 16 + React 19 (existing VentureX app) · **Status:** Spec ready, not built · **Related:** `M15_DESIGN.md`, `M15_SPRINT_PLAN.md`, the per-candidate dossier page at `/ventures/[id]/dossier/[candidate]`

---

## 1. Purpose

We have per-candidate dossiers working (52 cells × N candidates, with tri-state confidence and Exa citations). We now need an **aggregate side-by-side view** — parameters as rows, candidates as columns — that:

1. Demos cleanly to Daniel as the M15 deliverable and the side-by-side comparison against Competely.
2. Lets a consultant scan 5 candidates at once, drill into any cell, and verify citations in <5 seconds.
3. Exports to a single self-contained HTML file that can be emailed to non-VentureX users (Daniel, DPZ, future clients) without losing interactivity.

The viewer is the **demo gate** artifact. Competely's equivalent (see `Analysis__Figma_vs_XD_vs_Webflow_vs_Canva__Competely.pdf` in project knowledge) is the explicit bar we want to beat — better cell density, real citations, honest gap handling, faster scanning.

---

## 2. Deliverable

Two artefacts, same UI:

**A. In-app page** at `/ventures/[id]/table` — reads live data from the `cells` table, renders all candidates for that venture in one view. Linked from `/ventures/[id]`.

**B. Standalone HTML export** — single self-contained `.html` file with data embedded inline. Opens in any browser, works offline, no external requests beyond citation links. Generated by a "Download HTML" button on the in-app page. Filename pattern: `venturex_{venture_slug}_{YYYYMMDD}.html`.

Both share the same React component tree. The export is generated by server-rendering the page with data embedded as `<script type="application/json" id="venturex-data">`, plus inlined CSS and a small vanilla-JS hydration layer for the modal interactions (no React runtime needed in the export).

## 3. Data Inputs

Source: the `cells` table from migration 0007 (M15-T1), joined with `candidates` and the parameter schema from `parameter_generation_runs.full_parameter_schema`.

### Cell shape (already produced by M15 orchestrator)

```ts
{
  candidate_id: string,           // UUID
  parameter_key: string,           // snake_case, e.g. "busbar_tap_off_offering"
  parameter_label: string,         // human-readable, e.g. "Busbar + tap-off unit offering"
  tier: 1 | 2 | 3,
  confidence: "verified" | "inferred" | "unknown",
  value: string | object | array | null,
  citations: Citation[],           // [] for T1 training-data cells, 1+ for T2/T3
  reason: string | null,           // present when confidence === "unknown"
                                   // e.g. "no_evidence_found", "extraction_error: ...",
                                   // "Supplied M13 citations do not identify ..."
  retrieved_at: string | null      // ISO timestamp, when applicable
}

Citation = {
  source_title: string,            // e.g. "APC NetShelter Advanced Rack PDUs | Schneider Electric USA"
  url: string,
  snippet: string,                 // ~1-3 sentences of source content
  retrieved_at: string             // ISO timestamp
}
```

### Candidate shape

```ts
{
  candidate_id: string,
  name: string,                    // e.g. "Schneider Electric"
  product_line: string | null,     // e.g. "APC NetShelter" (rendered in parens if present)
  logo_url: string | null,
  stats: {
    total: number,
    verified: number,
    inferred: number,
    unknown: number
  }
}
```

### Parameter schema shape

Drawn from `parameter_generation_runs.full_parameter_schema`. Parameters are pre-ordered within each tier; the viewer respects that order.

```ts
{
  parameter_key: string,
  parameter_label: string,
  tier: 1 | 2 | 3,
  description: string,             // shown on hover over the parameter label
  value_shape: "string" | "number" | "object" | "array"  // hint for cell renderer
}
```

## 4. Layout

### 4.1 Orientation

**Parameters as rows, candidates as columns.** Matches Competely's layout and the user's mental model. Left column holds parameter labels + tier badges; top row holds candidate headers.

### 4.2 Frozen header row and column

Both must stick on scroll, in both axes:

- **Top row** (candidate headers) stays fixed when scrolling vertically.
- **Left column** (parameter labels) stays fixed when scrolling horizontally.
- The **top-left corner cell** stays fixed in both directions and holds the venture title + cells-count summary (e.g., "ABB Rack PDU — 260 cells across 5 candidates").

Implementation: CSS `position: sticky` with `top: 0` for the row, `left: 0` for the column, and `top: 0; left: 0; z-index: 3` for the corner. Confirm cross-browser behaviour on Safari (a common sticky-table gotcha).

### 4.3 Tier sectioning

Parameters are grouped by tier. Each tier opens with a **tier section header row** spanning all candidate columns:

```
┌──────────────────────────────────────────────┐
│ Tier 1 — Universal (15 parameters)           │ ← section header
├──────────────────────────────────────────────┤
│ Legal Name      │ Schneider │ Eaton │ ...    │
│ Founded Year    │ Schneider │ Eaton │ ...    │
│ ...                                          │
├──────────────────────────────────────────────┤
│ Tier 2 — Framework (21 parameters)           │ ← section header
├──────────────────────────────────────────────┤
│ ...                                          │
└──────────────────────────────────────────────┘
```

Tier section headers are collapsible — click to fold/unfold the whole tier. Default state: all expanded. Tier section headers also stick to the top when scrolling within their section (secondary sticky behaviour; if it complicates implementation, drop it).

### 4.4 Dimensions

- **Parameter column** (left): 240px wide.
- **Candidate columns**: 280px wide each. Horizontal scroll past 4 candidates on a 1440px screen.
- **Row height**: minimum 96px, grows to fit summary content.
- **Header row**: 96px (logo 40px + name + stats).
- **Tier section header**: 48px.

### 4.5 Cell summary view

Inside each non-header cell:

- **Top-right corner**: confidence indicator — coloured dot (8px) with tooltip.
- **Body**: type-based summary rendering (see §5).
- **Bottom-right** (only if citation exists): small link icon indicating citation available.
- **Click target**: entire cell. Cursor `pointer` on hover. Hover state: subtle background lift (`--bg-cell-hover`).

### 4.6 Modal expanded view

Click any cell → modal opens. The modal contains:

- **Header**: parameter label (full text) · tier badge · confidence badge · candidate name.
- **Value block**: full content. If JSON, render as a key-value list (not raw JSON unless toggled). If array, render as bullets. If string, render as prose.
- **Citation block(s)**: one per citation. Each shows source title, clickable URL (opens in new tab, `rel="noopener noreferrer"`), `retrieved_at` formatted as "retrieved 2026-05-20", and snippet in a styled blockquote.
- **For T1 cells with no citation**: instead of citation block, show a small italic note: *"Training-data value — no citation required for Tier 1 identity facts."*
- **For unknown cells**: instead of citation block, show the `reason` field in a styled callout (e.g., `no_evidence_found` rendered as "No evidence found after broadening retry.").
- **For extraction errors**: render the error reason verbatim with an error icon.
- **Footer**: copy-to-clipboard button for the parameter key (useful when filing issues), and a close button.

Modal dismissal: X button (top-right), `Esc` key, click on backdrop. Focus trap inside the modal while open; restore focus to the originating cell on close.

## 5. Cell Summary Rendering by Value Type

The cell summary is the at-a-glance view inside the table. Keep summaries terse — 2-4 lines max, truncate with ellipsis if overflowing. Full content lives in the modal.

| Value type | In-cell summary |
|---|---|
| `string` (short, <120 chars) | Render as-is. |
| `string` (long) | First sentence, ellipsis. |
| `number` | Format with thousands separators; for currency, prefix symbol. |
| `object` (small, ≤3 keys) | Key-value list, one per line: `key: value`. |
| `object` (large) | 2-3 most important fields (defined in parameter schema as `summary_keys`, falling back to first 2 keys). |
| `array of strings` | First 3 items as inline bullets (`• item • item • item`). If more, append `+N more` in muted text. |
| `array of objects` | Count + first item's primary field: `3 events — most recent: 2023 AVEVA acquisition`. |
| `null` (with reason) | Render `reason` as muted italic text. Don't say "unknown" — say the actual reason. |

### 5.1 Confidence indicator colours

Use existing `DESIGN.md` design tokens if present. Otherwise:

- **Verified** — green (`--confidence-verified`, suggested `#059669`).
- **Inferred** — amber (`--confidence-inferred`, suggested `#d97706`).
- **Unknown** — grey (`--confidence-unknown`, suggested `#9ca3af`).

The dot is the only colour on the cell. Avoid filling the whole cell with the confidence colour — keep the table calm, let the data speak.

### 5.2 Tier badges

Small uppercase tags next to the parameter label in the left column: `T1`, `T2`, `T3`. Each tier gets a different muted border colour (no fill). The badge order follows the tier numerical order.

## 6. Visual Design

### 6.1 Aesthetic

**Refined minimalism.** This is a tool consultants use for analytical work — the UI must not compete with the data. Generous whitespace, restrained palette, no decorative gradients or flashy hovers. The citations and confidence indicators are the only colour; everything else is greyscale.

Avoid the generic-SaaS look: no purple gradients, no excessive shadows, no rounded-12px-everywhere. Borders are 1px solid; corners are 4px max. Think the New York Times opinion section, not a typical landing page.

### 6.2 Typography

- **Body**: a refined sans-serif. Defer to existing app typography choices. If choosing fresh, prefer something with character (e.g., `Söhne`, `Inter Display`, `GT America`) over `Inter`/`Roboto`/system-ui.
- **Cell summary**: 13px, 1.45 line-height.
- **Parameter labels**: 14px, medium weight.
- **Candidate names**: 15px, semibold.
- **Modal body**: 15px, 1.6 line-height.
- **Citations**: 13px, monospace-tinged sans for the URL line.

### 6.3 Spacing and density

- Cell padding: 12px vertical, 14px horizontal.
- Section header: 16px vertical, 14px horizontal.
- Modal padding: 24px.
- Modal max-width: 720px; max-height: 80vh; scrolls internally if content overflows.

### 6.4 Header row content

For each candidate header cell:

```
┌──────────────────────────────────┐
│ [logo]  Schneider Electric       │
│         APC NetShelter           │  ← product_line in muted text if present
│         52 · 15v · 21i · 16u     │  ← stats line, muted
└──────────────────────────────────┘
```

The stats line uses single-letter shorthand (`v` verified, `i` inferred, `u` unknown). Click the stats line → modal opens with a per-candidate summary (out of scope for V1; reserve the click target).

### 6.5 Top-left corner cell

```
┌──────────────────────────────────┐
│ VentureX                         │
│ ABB Rack PDU                     │
│ 5 candidates · 260 cells         │
│ generated 2026-05-20             │
└──────────────────────────────────┘
```

Subtle border on right and bottom only.

## 7. Technical Requirements

### 7.1 In-app page

- Route: `app/ventures/[id]/table/page.tsx`.
- Server component fetches cells + candidates + parameter schema; passes to client component.
- Client component handles modal state and tier-collapse state.
- No new dependencies beyond what M15 already brings in.

### 7.2 Standalone HTML export

- Single `.html` file. CSS inlined in `<style>`, JS inlined in `<script>`, data embedded in `<script type="application/json" id="venturex-data">`.
- No external requests except the citation links (which are external by definition).
- No CDN fonts — fall back to system fonts gracefully if custom fonts can't be inlined as base64 (custom fonts are *nice-to-have*; system fallback must remain clean).
- File size target: <500KB for a 5-candidate × 51-parameter export. Larger only if citations include long snippets.
- Browser support: latest Chrome, Safari, Firefox. No IE11.
- The export is generated by an API route, e.g. `GET /api/ventures/[id]/table/export` returning HTML with a `Content-Disposition: attachment` header.

### 7.3 Data swap point

For the standalone export, the data is identifiable by the `id="venturex-data"` script block:

```html
<script type="application/json" id="venturex-data">
{
  "venture": {...},
  "candidates": [...],
  "parameters": [...],
  "cells": [...]
}
</script>
```

This lets us hand-edit the file or regenerate it from a script without touching the rendering layer.

### 7.4 Performance

- Initial render: <500ms for 5 candidates × 51 parameters = 255 cells.
- Smooth scroll in both axes; no jank when sticky elements update.
- Modal open: <100ms perceived (no fetch — all data is already in memory).
- Use `content-visibility: auto` on cell content if scroll jank appears.

## 8. Edge Cases

These have all surfaced in the Schneider first-run output (see `VentureX.pdf` in project knowledge). Handle them deliberately rather than hoping they don't appear.

| Case | Spec |
|---|---|
| Cell missing entirely (no row in `cells` table for this candidate × parameter) | Render the cell with a muted dash `—` and tooltip "Not researched". This should be rare once the orchestrator is reliable; treat it as a data anomaly worth surfacing. |
| `confidence: "unknown"` with `reason: "no_evidence_found"` | Cell summary: muted italic "No supporting evidence found." Modal: full reason text in a callout. |
| `confidence: "unknown"` with `reason: "extraction_error: ..."` | Cell summary: muted "Extraction failed." Modal: error icon + full reason. Flagged visually as different from "no evidence" (a system failure, not an honest gap). |
| `confidence: "unknown"` with `reason: "Supplied M13 citations do not identify ..."` | Cell summary: muted italic "Gap in M13 evidence." Modal: full reason. (This pattern is a known T2 issue documented separately; consistent rendering makes the gap visible without looking broken.) |
| Tier 1 cell with `citations: []` | Normal rendering. Modal shows the training-data note. **This is correct by design and must not look like a failure.** |
| Value is genuinely null (e.g., `parent_company` for Schneider — it has no parent, it IS one) | Display as "None" not "Unknown". Requires the orchestrator to distinguish; if it currently doesn't, this edge case is informational for the next pipeline iteration. |
| Long nested JSON (e.g. `funding_ma_history` with 4 entries) | Cell summary: count + most-recent summary. Modal: pretty-printed full list. |
| Very long string (>500 chars in cell) | Summary: first sentence + ellipsis. Modal: full text. |
| Logo missing | Render a 40px square with the candidate's first letter in muted text. |
| 10+ candidates | Horizontal scroll works as expected; performance still <500ms. Out of immediate scope but don't architect against it. |

## 10. Acceptance Criteria

The Claude Code agent should treat these as a pass/fail checklist. Don't ship until every box checks.

1. **Sticky behaviour works in both axes** — top row stays put on vertical scroll; left column stays put on horizontal scroll; corner cell stays put in both. Tested in Chrome and Safari.
2. **Tier sections render with collapsible headers**, all expanded by default.
3. **Cell summary renders correctly for every value shape** in §5 (string, number, object small/large, array of strings, array of objects, null with reason).
4. **Confidence indicator is visible on every cell** and the colour matches the tier value (verified/inferred/unknown).
5. **Clicking any cell opens the modal** with the matching cell's full content, citations, and reason (if applicable).
6. **Modal closes on X, Esc, and backdrop click**. Focus returns to the originating cell.
7. **Citation URLs open in a new tab** with `rel="noopener noreferrer"`.
8. **T1 cells with no citations render normally** and the modal shows the training-data note — explicitly *not* styled as a failure.
9. **Unknown cells distinguish between `no_evidence_found`, `extraction_error`, and "M13 gap"** in the cell summary text. All three render without looking broken.
10. **Standalone HTML export downloads as a single file**, opens correctly with no external requests besides citation links, and renders identically to the in-app page (modulo any in-app interactive features that don't apply to the export).
11. **Render time <500ms** for the 5-candidate × 51-parameter case (255 cells). Verify with `performance.now()` in the browser console.
12. **Beats the Competely benchmark** on a side-by-side spot-check: open the Competely PDF and the new viewer for any one parameter. The new viewer should show specific cited facts where Competely shows generic prose. (Subjective — flag any cases where this isn't true.)

## 11. Out of Scope (V2 / V1.x)

To prevent feature creep — these will tempt the agent, none of them ship in V1:

- Cell editability or re-research from the table view (per `M15_DESIGN.md`, V1 is read-only verification).
- Cross-candidate comparison annotations ("here's how these 5 differ on `pricing_disclosure`") — interesting V2 feature but a different research pass.
- Filtering or sorting parameters (just render in tier order).
- Multi-venture comparison (one venture per view).
- Export to PDF or .xlsx (HTML export only for V1).
- Real-time updates (the table is a snapshot at generation time).
- Mobile-first responsive design — the viewer is desktop-primary; mobile readable but not optimised.
- Search across cells.
- Saved views or shareable URL state.

## 12. Open Questions

Surface these explicitly so the agent doesn't paper over them:

1. **Custom typography:** If the existing app uses `Inter`, leave it; if there's room to choose a more characterful body font (e.g., `Söhne`, `GT America`), use it. Confirm with Harry before pulling in any new font dependency.
2. **Per-candidate summary modal** (clicking the stats line in a candidate header): reserved as a click target but content TBD. V1 can no-op; V2 fills in.
3. **HTML export server-side rendering:** confirm the Next.js setup supports rendering the page to an HTML string for export. If not, the agent may need to build a minimal duplicate renderer for the export path. Estimate: ~1h either way.
4. **Logo source:** the `candidates.logo_url` field isn't yet populated by the M13 stage. For the demo, hand-curate logos for the 5-candidate shortlist or fall back to the letter avatar.
5. **"None" vs "Unknown" distinction** (§8): does the orchestrator currently emit `value: null, reason: null` for cells that are genuinely null (Schneider has no parent), or does everything null land as `confidence: "unknown"`? If the latter, this needs a small upstream fix or a special-case in the renderer. The first-run dossier suggests the latter — flag for Harry.
